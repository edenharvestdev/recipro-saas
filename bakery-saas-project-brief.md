# Recipe-Cost SaaS — เอกสารโปรเจกต์ฉบับสมบูรณ์ (สำหรับ Claude Code)
### ระบบคิดต้นทุน + ขาย สำหรับธุรกิจที่มี "สูตร" (ร้านเล็ก/ทำคนเดียว) — หลายร้าน + สมาชิกรายเดือน/ปี + แอดมินหลัก

> ⚠️ **อัปเดตทิศทาง (ล่าสุด):** เอกสารนี้เป็นบันทึกประวัติ/เหตุผลเชิงเทคนิคช่วงแรก เนื้อหาด้านล่างยังอ้างถึง
> Supabase / Vercel / Stripe ซึ่ง **เลิกใช้แล้ว** สแตกที่ใช้จริงตอนนี้คือ **Railway + Node/Express + PostgreSQL + JWT + Omise**
> แหล่งอ้างอิงที่ถูกต้อง: `docs/architecture.md` · `docs/migration-plan.md` · `docs/marketing-pricing.md`
> (อ่านส่วนล่างได้เพื่อเข้าใจเหตุผล/ฟีเจอร์ธุรกิจ แต่เรื่อง "สแตก/โฮสต์/จ่ายเงิน" ให้ยึดเอกสารใหม่)

> **จุดยืน:** ระบบนี้ไม่จำกัดแค่เบเกอรี — ใช้ได้กับทุกธุรกิจที่ผลิตของจาก "วัตถุดิบ + สูตร" แล้วต้องคิดต้นทุน (ดูกลุ่มลูกค้าเป้าหมายข้อ 1.1) เบเกอรี Merry Jane เป็นเพียงร้านตัวอย่างร้านแรก

> **วิธีใช้เอกสารนี้:** สร้างโฟลเดอร์โปรเจกต์ใหม่ 1 โฟลเดอร์ วางไฟล์นี้ + ไฟล์ `merryjane.html` (หรือ `index.html`) + `bakery-system.html` ลงไป แล้วเปิด Claude Code ในโฟลเดอร์นั้น สั่งให้ทำตามเฟสในข้อ 8 ทีละเฟส

## 1. สรุปสำหรับเจ้าของ (อ่านง่าย)

ตอนนี้เรามี **แอปคิดต้นทุนจากสูตร** ที่ทำงานครบ (คิดต้นทุน, ออกบิล, สต๊อก/ผลิต, โคลนสูตร ฯลฯ) แต่เป็น **ไฟล์เดียวต่างคนต่างใช้** ขายต่อแบบเก็บเงินรายเดือนไม่ได้

โปรเจกต์นี้คือการยกระดับเป็น **"ระบบจริง" (SaaS)** ที่:
- มี **ระบบล็อกอิน** ลูกค้าแต่ละร้านมีไอดีของตัวเอง
- **คุณเป็นแอดมินหลัก (super admin)** เปิดร้านใหม่ / ปิดร้าน / ดูทุกร้านได้
- ร้าน **Merry Jane เดิม = หนึ่งในร้านในระบบ** (ร้านแรกของคุณเอง)
- เก็บ **ค่าสมาชิกรายเดือน/รายปี** + **ตัดบัตรเครดิตอัตโนมัติ** + **ต่อสัญญาเอง**
- **อัปเดตที่เดียว ทุกร้านได้พร้อมกัน**

ผลลัพธ์: คุณขายเป็นบริการรายเดือนได้จริง ลูกค้าสมัคร–จ่าย–ใช้งานได้เอง

### 1.1 กลุ่มลูกค้าเป้าหมาย (ขายได้กว้างกว่าเบเกอรี)

หัวใจของระบบคือ **"วัตถุดิบ → สูตร → ต้นทุน → ราคาขาย → ออกบิล → สต๊อก"** จึงเหมาะกับ **ทุกธุรกิจเล็ก/ทำคนเดียวที่ผลิตของจากส่วนผสมหลายตัว** เช่น:
- เบเกอรี / เค้ก / ขนม
- ร้านกาแฟ / เครื่องดื่ม / ชานมไข่มุก / น้ำผลไม้
- ร้านอาหารตามสั่ง / ครัวโฮมเมด / catering / อาหารกล่อง
- สบู่ / เทียนหอม / เครื่องสำอาง / สกินแคร์ DIY
- น้ำหอม / น้ำยาทำความสะอาด / ผลิตภัณฑ์ผสมเอง
- งานคราฟต์ที่มีต้นทุนวัตถุดิบหลายตัว

**ผลต่อการออกแบบระบบ:** คำที่โชว์ให้ผู้ใช้ควรเป็นกลาง — ใช้ "สินค้า/เมนู" แทน "ขนม", "ส่วนผสม/วัตถุดิบ" แทนคำเฉพาะเบเกอรี เพื่อให้ทุกธุรกิจใช้ได้ (อาจมี "ประเภทธุรกิจ" ให้เลือกตอนเปิดร้าน เพื่อปรับคำ/ตัวอย่างให้เข้ากับสายงาน — ทำทีหลังได้)

## 2. เป้าหมายโปรเจกต์ (Requirements)

| # | ความต้องการ | รายละเอียด |
|---|---|---|
| R1 | ระบบล็อกอิน | สมัคร/เข้าสู่ระบบด้วยอีเมล+รหัส (หรือ magic link) |
| R2 | หลายร้าน (multi-tenant) | ข้อมูลแต่ละร้านแยกขาด ร้าน A ไม่เห็นร้าน B |
| R3 | Merry Jane เป็นร้านหนึ่ง | ย้ายข้อมูลเดิม (วัตถุดิบ/ซัพพลายเออร์/สูตร) เข้าระบบเป็นร้านแรก |
| R4 | แอดมินหลัก (super admin) | คุณเปิด/ปิด/ดู/แก้ทุกร้านได้จากแผงเดียว |
| R5 | สมาชิกรายเดือน/รายปี | มีแพ็กเกจราคา เลือกจ่ายรายเดือนหรือรายปี |
| R6 | ตัดบัตรเครดิตอัตโนมัติ | เก็บเงินซ้ำตามรอบ โดยลูกค้าอนุมัติบัตรไว้ล่วงหน้า |
| R7 | ต่อสัญญา / หมดอายุ | ต่ออัตโนมัติ, แจ้งเตือนก่อนหมด, บัตรเด้ง→พักร้าน |
| R8 | ใช้ฟีเจอร์เดิมทั้งหมด | คิดต้นทุน, บิล 3 แบบ, สต๊อก/ผลิต, วิธีทำ, โคลนสูตร, แบรนด์/ธีม |

## 3. ของเดิมที่มีแล้ว — นำกลับมาใช้ (ลดงาน/ลดต้นทุนมาก)

ไฟล์ `merryjane.html` / `bakery-system.html` **ทำตรรกะธุรกิจครบแล้ว** ได้แก่:
- คำนวณต้นทุนต่อหน่วย + แปลงหน่วยวัตถุดิบ (กก./กรัม/ลิตร/มล./ฟอง/แพ็ค/ขวด)
- จัดการซัพพลายเออร์ + ลิงก์สั่งซื้อ (Shopee) + นำเข้า Excel/CSV
- สูตร: ราคาขาย, จำนวนต่อครั้ง + หน่วย, วิธีทำ, **โคลนสูตร**, Raw material (สูตรเป็นวัตถุดิบในสูตรอื่น)
- ออกบิล 3 แบบ (ใบส่งสินค้า/ใบวางบิล/ใบเสร็จ) + ส่วนลด/ภาษี + สถานะ + PromptPay
- สต๊อก/ผลิต: รับวัตถุดิบ, แจ้งเตือนของใกล้หมด, ผลิต (ตัดวัตถุดิบ + เพิ่มสินค้าสำเร็จ), ตัดสต๊อกตอนออกบิล
- แบรนด์/ธีม: อัปโหลดโลโก้ + เลือกธีมสี 5 แบบ (มีใน `bakery-system.html`)
- สำรอง/กู้คืนข้อมูลเป็นไฟล์ JSON

**จุดต่อสำคัญสำหรับ Claude Code:** แอปเก็บข้อมูลผ่าน shim กลางชื่อ `DB.get() / DB.set()` (ตอนนี้ชี้ไป `localStorage`). งานหลักของการทำ SaaS คือ **เปลี่ยน implementation ของ `DB` ให้เรียกฐานข้อมูลกลาง (Supabase) แทน** + ผูกทุก query กับ `shop_id` + ครอบด้วยหน้า login ไม่ต้องเขียน UI ธุรกิจใหม่

## 4. สถาปัตยกรรม (Architecture)

```
[ผู้ใช้ร้านค้า] ──┐
[Merry Jane]   ──┤   เว็บแอปตัวเดียว (Frontend)         โฮสต์: Vercel / Netlify
[แอดมินหลัก]    ──┘   ต่อยอดจาก bakery-system.html         (อัปเดตที่เดียว = ทุกคนได้)
                         │
                         ├──► Supabase ── Auth (ล็อกอิน)
                         │              ── Database (PostgreSQL) ทุกร้านอยู่ที่เดียว
                         │              ── Row Level Security (RLS) แยกข้อมูลแต่ละร้าน
                         │              ── Edge Functions (รับ webhook จากระบบจ่ายเงิน)
                         │
                         └──► ระบบจ่ายเงิน ── Stripe Billing  หรือ  Omise/Opn (ไทย)
                                            (subscription + ตัดบัตรอัตโนมัติ + ต่อสัญญา)
```

## 5. เทคโนโลยีที่แนะนำ (Tech stack)

| ส่วน | แนะนำ | หมายเหตุ |
|---|---|---|
| Auth + Database + RLS | **Supabase** (PostgreSQL) | ได้ login + ฐานข้อมูล + แยกข้อมูลหลายร้าน ในที่เดียว เหมาะกับข้อมูลเชิงสัมพันธ์ |
| Frontend | ต่อยอดโค้ดเดิม (HTML/JS) หรือรีแฟกเตอร์เป็น SPA เล็ก ๆ | ตรรกะธุรกิจมีแล้ว |
| Hosting | **Vercel** หรือ **Netlify** | รองรับ serverless function สำหรับ webhook (GitHub Pages เป็น static อย่างเดียว ใช้ทำ frontend ได้ แต่ต้องมีที่รัน webhook แยก เช่น Supabase Edge Functions) |
| ระบบจ่ายเงิน | **Stripe Billing** หรือ **Omise/Opn Payments (ไทย)** | ดูข้อ 7 |

> ทางเลือก backend อื่น: Firebase (Firestore + Auth) ก็ได้ แต่ข้อมูลเราเป็นเชิงสัมพันธ์ Supabase/SQL ตรงกว่า — ให้ dev/Claude Code ตัดสินใจสุดท้าย

## 6. โครงสร้างฐานข้อมูล (Database schema)

ทุกตารางข้อมูลร้านมี `shop_id` เพื่อแยกร้าน

```sql
-- ผู้ใช้ & ร้าน & สิทธิ์
shops          (id, name, status, created_at)          -- status: active | suspended | trial
memberships    (id, user_id, shop_id, role)            -- role: superadmin | owner | staff
shop_settings  (shop_id, phone, tax_id, address, bank, account,
                holder, promptpay, logo_url, theme)

-- ข้อมูลธุรกิจ (มาจากแอปเดิม — มี shop_id ทุกตาราง)
suppliers      (id, shop_id, name, note)
materials      (id, shop_id, name, qty, unit, price, supplier_id,
                order_url, stock, low_stock)
recipes        (id, shop_id, code, name, sell_price, batch_yield,
                yield_unit, is_raw, steps, fg_stock, fg_low)
recipe_items   (id, recipe_id, material_id, amount)
bills          (id, shop_id, number, doc_type, items_json,
                discount, tax, status, stock_deducted, created_at)

-- ระบบสมาชิก/จ่ายเงิน (ใหม่)
plans          (id, name, price_month, price_year, features_json, active)
subscriptions  (id, shop_id, plan_id, status, billing_cycle,
                current_period_end, cancel_at,
                provider, provider_customer_id, provider_sub_id)
                -- status: trialing | active | past_due | canceled | suspended
payments       (id, shop_id, amount, currency, status, paid_at,
                provider_invoice_id)                    -- เก็บประวัติการจ่าย (ไม่เก็บเลขบัตร!)
```

**สำคัญด้านความปลอดภัย:** ห้ามเก็บเลขบัตรเครดิตในฐานข้อมูลเราเอง — ให้ Stripe/Omise เก็บแบบ token เท่านั้น (ดูข้อ 10)

## 7. ระบบสมาชิก + ตัดบัตรอัตโนมัติ (Billing)

ทั้งสองเจ้าใช้ในไทยได้และรองรับ subscription/ตัดบัตรซ้ำ — เลือกเจ้าใดเจ้าหนึ่ง:

**ตัวเลือก A — Stripe Billing**
- มี Subscriptions, recurring payments, Checkout, Invoicing, ลองเก็บเงินซ้ำเมื่อบัตรเด้ง (Smart Retries) ครบในตัว
- เอกสาร/ตัวอย่างเยอะ เชื่อมกับ Supabase ง่าย (มี template สำเร็จ)
- เหมาะถ้าอยากโตข้ามประเทศในอนาคต

**ตัวเลือก B — Omise / Opn Payments (เจ้าของไทย)**
- ทำ recurring ได้ผ่าน **Schedule API** + เก็บบัตรแบบ token-on-file (PCI DSS), รองรับบัตร + PromptPay + ธนาคาร
- เป็นเจ้าในไทย คุยซัพพอร์ตภาษาไทยได้ ลูกค้าคุ้นเคย
- ข้อจำกัดเล็กน้อย: ตารางตัดเงินรายเดือน **ไม่รองรับวันที่ 29–31** (ตั้งรอบเป็นต้นเดือนแทน)

> เจ้าอื่นในไทยที่ทำได้: 2C2P, Pay Solutions, TrueMoney — ไว้เทียบราคาทีหลังได้

**โฟลว์สมาชิก (เหมือนกันทั้งสองเจ้า):**
1. ลูกค้าสมัคร → เลือกแพ็กเกจ (รายเดือน/รายปี) → กรอกบัตร (หน้าจ่ายเงินของ Stripe/Omise โดยตรง เราไม่แตะเลขบัตร)
2. ระบบสร้าง subscription + ตัดเงินรอบแรก → `subscriptions.status = active` → เปิดใช้ร้าน
3. ถึงรอบถัดไป ระบบตัดบัตรอัตโนมัติ → ส่ง **webhook** มาอัปเดตสถานะ
4. บัตรเด้ง → ลองใหม่อัตโนมัติ + แจ้งเตือนลูกค้า → ถ้ายังไม่จ่ายในเวลาผ่อนผัน (เช่น 7 วัน) → `status = suspended` → ร้านเข้าได้แต่ใช้ไม่ได้จนกว่าจะจ่าย
5. ยกเลิก/ต่อสัญญา: ลูกค้าจัดการเองผ่านหน้า portal ของผู้ให้บริการ

## 8. แผนการสร้างเป็นเฟส (สั่ง Claude Code ทีละเฟส)

> ตัวเลขเวลาเป็น **ค่าประมาณคร่าว ๆ** ขึ้นกับความเร็วการทำงานจริง

**เฟส A — รากฐาน (Auth + Multi-tenant + ย้ายแอปเดิม)** ~1–2 สัปดาห์
- ตั้ง Supabase + สร้าง schema ข้อ 6 (ส่วนข้อมูลธุรกิจ) + เปิด RLS
- ทำหน้า login + memberships + role
- เปลี่ยน `DB` shim จาก localStorage → Supabase + ผูก `shop_id`
- ย้ายข้อมูล Merry Jane เข้าเป็นร้านแรก (จากไฟล์สำรอง JSON)

**เฟส B — แอดมินหลัก (Super admin)** ~3–5 วัน
- แผงแอดมิน: สร้างร้านใหม่, เชิญผู้ใช้, เปิด/ปิด/พักร้าน, ดูทุกร้าน
- ผูกแบรนด์/ธีม/โลโก้ ต่อฐานข้อมูล (UI มีแล้วใน bakery-system.html)

**เฟส C — สมาชิก + ตัดบัตร (Billing)** ~1–2 สัปดาห์
- ตั้งแพ็กเกจ (รายเดือน/รายปี) ใน Stripe/Omise + ตาราง plans/subscriptions/payments
- หน้าเลือกแพ็กเกจ + หน้าจ่ายเงิน + webhook อัปเดตสถานะ
- ลอจิกต่อสัญญา/หมดอายุ/บัตรเด้ง/พักร้าน + แจ้งเตือนอีเมล

**เฟส D — เก็บงาน** ~3–5 วัน
- หน้าแดชบอร์ดแอดมิน (รายได้/จำนวนร้าน/ร้านใกล้หมดอายุ)
- อีเมลแจ้งเตือน, ใบเสร็จค่าบริการ, ทดสอบ end-to-end, deploy

**รวม MVP (A–D):** ประมาณ **4–7 สัปดาห์**

## 10. ความปลอดภัย & ข้อกฎหมาย (อย่าข้าม)

- **อย่าเก็บเลขบัตรเครดิตเองเด็ดขาด** — ส่งให้หน้าจ่ายเงินของ Stripe/Omise โดยตรง แล้วเก็บแค่ "token/รหัสอ้างอิง" (PCI DSS) เราจึงไม่ต้องรับภาระความปลอดภัยบัตรเอง
- **PDPA (พ.ร.บ.คุ้มครองข้อมูลส่วนบุคคล)** — มีหน้านโยบายความเป็นส่วนตัว + ขอความยินยอมเก็บข้อมูล
- **RLS เปิดทุกตาราง** — กันร้านหนึ่งเห็นข้อมูลอีกร้านเด็ดขาด (หัวใจของ multi-tenant)
- เก็บรหัสผ่าน/คีย์ลับไว้ใน environment variables ไม่ฝังในโค้ด frontend

## 13. แผนอนาคต (Roadmap หลัง MVP)

- **เฟส E — HQ → หลายสาขา:** ร้านเดียวมีหลายสาขา, สต๊อกแยกสาขา, แจ้งเตือนข้ามสาขา, รายงานรวม
- **เฟส F — รายงาน/วิเคราะห์:** กำไรต่อเมนู, ยอดขายรายเดือน, วัตถุดิบที่ใช้บ่อย
- **เฟส G — แอปมือถือ (PWA/เนทีฟ):** ติดตั้งหน้าจอโฮม, แจ้งเตือน push
- **เฟส H — ตลาด/ลูกค้า:** หน้าเว็บขาย, สมัครเอง, โค้ดส่วนลด, ระบบแนะนำเพื่อน
- **เฟส I — ปรับตามสายธุรกิจ:** เลือก "ประเภทธุรกิจ" ตอนเปิดร้าน (เบเกอรี/กาแฟ/อาหาร/สบู่-เครื่องสำอาง/อื่นๆ) เพื่อปรับคำศัพท์ + เมนูตัวอย่าง + หน่วยวัด ให้เข้ากับแต่ละสาย
