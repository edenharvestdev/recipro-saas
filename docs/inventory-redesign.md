# Inventory & Recipe Redesign — Review + Migration Plan

> สถานะ: **review only — ยังไม่แก้โค้ดระบบ** รอเจ้าของเคาะทิศทาง + คำถามฐานราก
> เป้าหมาย: Item Master + Recipe Layer, ตัดสต๊อกตามประเภท, คิดต้นทุนหลายชั้น — โดย **เก็บข้อมูลร้านเดิมครบ** ให้ร้านไปจัดประเภทเอง

## 0. การตัดสินใจที่ล็อกแล้ว (จากเจ้าของ 2026-06-22)
- **8 หมวด = ชุดมาตรฐานกลาง** ใช้ได้กับทุกธุรกิจ เรียบง่ายสำหรับคนดูแลคนเดียว (ร้านเพิ่ม sub-tag เองได้ภายหลัง); workflow หนัก ๆ (เบิก/อนุมัติ) ตั้งเป็น "อนุมัติเอง" ได้สำหรับร้านเล็ก
- **Asset** อยู่ใน Item Master เดียว ติดธง `is_stock_deducted=false` ตอนนี้ — ออกแบบให้แยก asset register (ค่าเสื่อม) ได้ในเฟส ERP
- `item_category_id` **nullable**, เก็บ `legacy_category` ไว้ตลอด (เก็บข้อมูลเดิมครบ)
- **ยังไม่เริ่ม implement** — อยู่ในขั้นรีวิวแผน

## 0.1 หลักการใช้งาน (UX) — ง่ายทุกวัย · ยืดหยุ่น · ใช้จริง
หลักคิด: ระบบหลังบ้านมี 8 หมวด/สูตรหลายชั้น แต่ **คนใช้เห็นแค่สิ่งที่จำเป็น** ความซับซ้อนถูกซ่อน เปิดเมื่อพร้อม

1. **2 ถังก่อน ไม่ใช่ 8 หมวด** — ตอนเพิ่มของ ถามแค่ **"ขายให้ลูกค้า"** หรือ **"ใช้ทำของ"** (ปุ่มใหญ่ 2 ปุ่ม) หมวดละเอียด 8 แบบอยู่ใต้ "ระบุเพิ่ม (ไม่บังคับ)" — เดาให้อัตโนมัติ
2. **ภาษาคน ไม่ใช่ศัพท์เทคนิค** — เก็บ code ภายในนิ่ง แต่โชว์ชื่อไทยที่เข้าใจง่าย:
   วัตถุดิบ · ของผสมเอง · ของเตรียมไว้ · ของกึ่งสำเร็จ · ของขาย/เมนู · บรรจุภัณฑ์ · ของใช้สิ้นเปลือง · ของใช้ถาวร
3. **กรอกน้อยที่สุด** — บังคับแค่ ชื่อ + หน่วย + ราคา ที่เหลือเว้นได้ ไม่มี field สีแดงบังคับเยอะ ๆ
4. **เปิดทีละชั้น (progressive)** — Lv1 เพิ่มของ/ขาย/สต๊อกลด · Lv2 ผูกสูตร+เห็นต้นทุน · Lv3 ผลิต/yield/เบิก/waste/รายงาน
5. **ร้านเล็ก = ไม่มีพิธี** — การเบิก/อนุมัติ ตั้งเป็น "อนุมัติเอง" โดยปริยาย, daily auto-deduct ปิดไว้ก่อน เปิดเมื่ออยากใช้
6. **สูตรเป็นภาษาคน** — "เมนูนี้ใช้อะไรบ้าง?" เลือกของ+ใส่จำนวน ต้นทุนขึ้นสด; การใช้ของเตรียม (สูตรซ้อน) = แค่เลือกจาก list เดียวกัน ไม่โผล่คำว่า sub-recipe
7. **เข้าถึงได้ทุกวัย** — ตัวอักษรใหญ่ ปุ่มใหญ่ คอนทราสต์สูง ไอคอน+คำ (ไม่เอาไอคอนล้วน) ลบมี undo/ยืนยัน ใช้มือถือมือเดียวได้
8. **ยืดหยุ่นทุกธุรกิจ** — หมวดที่ไม่ใช้ซ่อนอัตโนมัติ (ร้านขายของแห้งใช้แค่ วัตถุดิบ/ของขาย/บรรจุภัณฑ์) · รองรับทั้ง "ทำสด" และ "ทำสต๊อกไว้"
9. **migration ใจดี** — หน้า "ช่วยจัดของให้เข้าที่": โชว์ของเดิม + หมวดที่เดาให้ + ปุ่ม "ใช่ ✓ / เปลี่ยน" ทำทีหลังได้ ข้ามได้ ไม่พังถ้าไม่ทำ

## 1. สถานะปัจจุบัน (ดีกว่าที่คิด)
- `materials` = ตารางสินค้า/วัตถุดิบเดียว มี: `category` (free text), `sale_type` (INGREDIENT_ONLY|SELLABLE), `is_consumable`, `sku`, `sell_price`, `conv_qty`/`stock_unit` (unit conversion)
- **สูตรซ้อนสูตรทำได้แล้ว**: `recipe_items.sub_recipe_id` → ชี้ไปสูตรอื่น (prep item เข้าเมนูได้), `recipes.is_sop` = ของกลาง
- `recipes`: `batch_yield`/`yield_unit` (yield), `fg_stock` (สต๊อกของที่ผลิตเสร็จ), `is_raw` (ไม่ถูกใช้จริงใน API)
- `stock_movements`: log การเคลื่อนไหว (`kind` = adjust/receive/produce/sale; cons-in/cons-out นิยามไว้แต่ไม่ใช้)
- `prod_logs` (รอบผลิต), `stock_receives` (รับเข้าบัตช์), options engine (ตัวเลือก/ขนาด)

## 2. ปัญหาราก (ทำไมต้องรื้อ)
1. **ไม่มี Item Master / ไม่มี taxonomy 8 หมวดจริง** — `category` เป็น free text ไม่บังคับ ทำให้ของซ้ำ/จัดผิด
2. **ประเภทสูตรไม่ชัด** — `is_raw` + `is_sop` กำกวม ไม่มี enum Production vs Menu
3. **ตัดสต๊อกไม่รู้ประเภท** — ทุก movement ใช้ `kind` รวม ๆ ไม่มีกฎ daily/manual/waste/packaging
4. **คิดต้นทุนอยู่ฝั่ง frontend** — ไม่มี engine หลังบ้านที่ roll-up ต้นทุนข้ามชั้นสูตร
5. **ตัดสต๊อกไม่ atomic กับบิล** — สร้างบิลกับตัดสต๊อกเป็นคนละ call
6. **Options ไม่ตัดสต๊อก** — เลือก oat milk +20ml แต่ไม่หักวัตถุดิบ
7. **ไม่มี API รายงานต้นทุน** — food cost / cost-per-menu / แยกหมวด

## 3. แนวทาง migrate — additive + เก็บข้อมูลครบ (หัวใจตามที่เจ้าของสั่ง)
**ไม่ลบ ไม่ทับ ไม่ยุ่งข้อมูลร้านที่ลงแล้ว** ทำแบบเพิ่มคอลัมน์ nullable + เปลี่ยนชื่อของเดิมเป็น `legacy_*`:

| ตาราง | เปลี่ยน | ผลต่อข้อมูลเดิม |
|---|---|---|
| `item_categories` (ใหม่) | สร้าง + seed 8 หมวดต่อร้าน | ตารางใหม่ ไม่กระทบของเดิม |
| `materials` | `category`→`legacy_category`; +`item_category_id` (nullable FK) | ค่าเดิมเก็บใน legacy_*, ใหม่เป็น NULL |
| `recipes` | `is_raw`→`legacy_is_raw`; +`recipe_type`(Production\|Menu); +`output_item_category_id` | ค่าเดิมคงไว้ |
| `stock_movements` | +`consumption_category` (nullable) | movement เดิม NULL |
| `bills` | +`effective_bom`, +`line_costs` (jsonb) | บิลเดิม NULL |
| `expenses` | +`cost_category` | ค่าเดิม NULL |
| ตารางใหม่ | `recipe_costing`, `cost_movements`, `manual_consumptions`, `waste_logs`, `daily_consumption_schedules` | ตารางใหม่ทั้งหมด |

ทุก migration **idempotent** (รันซ้ำทุก deploy ได้) โค้ดเก่าทำงานต่อได้จนร้านเลือกเปิดใช้ของใหม่

### Flow ให้ร้านจัดประเภทเอง
- หน้า dashboard เตือน "จัดหมวดสินค้าให้ครบเพื่อปลดล็อกรายงานต้นทุน"
- หน้า categorize: list สินค้าที่ยัง NULL + **auto-suggest หมวดจาก legacy_category** (fuzzy match) → "อนุมัติทั้งหมด" หรือแก้ทีละตัว + progress bar
- หน้า recipe: `is_raw` → เลือก Production/Menu + เลือก output category
- ปลดล็อกฟีเจอร์ขั้นสูงเมื่อจัดครบ ~80% (ที่เหลือเติมทีหลังได้, แก้ภายหลังได้เสมอ)

## 4. การแบ่งเฟส
- **Phase 0** — Schema (ปลอดภัย ไม่แตะข้อมูล): สร้างตาราง/คอลัมน์ใหม่ + seed 8 หมวด
- **Phase 1** — UI จัดหมวดสินค้า + auto-suggest
- **Phase 2** — จัดประเภทสูตร (Production/Menu + output category)
- **Phase 3** — Cost engine หลังบ้าน (roll-up ต้นทุนข้ามชั้น, nightly + on-demand)
- **Phase 4** — ตัดสต๊อกตามหมวด (packaging ตัดตอนขาย, ops ตัด daily/อนุมัติ) — opt-in
- **Phase 5** — รายงานต้นทุน (แยก Raw/Packaging/Ops/Waste/Production, food cost, cost/menu)
- **Phase 6** — Options ตัดสต๊อกจริง (effective BOM)
- **Phase 7** — workflow ขั้นสูง: daily auto-deduct, manual+อนุมัติ, waste, โอนสาขา

## 5. ความเสี่ยงหลัก + การกัน
- ของเยอะจัดไม่ไหว → auto-suggest + bulk + ค่อยเติมทีหลัง
- ย้ายคิดต้นทุนมาหลังบ้านอาจมี edge case (หน่วย/ปัด) → รันคู่ frontend สักระยะแล้วเทียบ
- เปลี่ยน API ตัดสต๊อก → เก็บ endpoint เก่าไว้, ของใหม่ opt-in
- สูตรซ้อนหลายชั้น → กันลูปวน (circular sub_recipe) + unit test 3 ชั้น
- cron daily ชนการปรับมือ → รันตี 6 + log ทุกการตัด + ปิดต่อ material ได้
