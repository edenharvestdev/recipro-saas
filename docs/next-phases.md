## เฟส J — ออกบิล/ใบเสร็จมาตรฐาน
เป้าหมาย: ยกระดับเอกสารให้ถูกต้องตามมาตรฐานไทย รองรับลูกค้าองค์กร/ขอใบกำกับภาษี

ขอบเขต
- ชนิดเอกสาร: ใบเสร็จรับเงิน · ใบกำกับภาษีเต็มรูป · ใบกำกับภาษีอย่างย่อ (POS) · ใบเสนอราคา/ใบส่งของ
- ฟิลด์ตามกฎหมาย: ชื่อ-ที่อยู่-เลขผู้เสียภาษี 13 หลัก (ผู้ขาย/ผู้ซื้อ), เลขที่เอกสารเรียงต่อเนื่องไม่ซ้ำ, วันที่, รายการ, VAT 7% แยกบรรทัด, ยอดก่อน/หลังภาษี
- ตัวเลือก: หัก ณ ที่จ่าย, ส่วนลด, โลโก้ + ตราประทับ + ลายเซ็น
- ส่งออก PDF (A4 + ใบเล็ก 80mm) พิมพ์ได้
- เลขรันเอกสารเก็บที่ `doc_counters` (ต่อร้าน/ปี/ชนิด) กันซ้ำ แบบ atomic

ข้อจำกัด/อนาคต: e-Tax Invoice & e-Receipt (ส่งกรมสรรพากร) ต้องผ่านผู้ให้บริการที่ ETDA รับรอง — แยกเป็นเฟสถัดไป

Prompt (Claude Code):
```
อ่าน backend/db/schema-extend.sql (bills + doc_counters)
ทำระบบออกเอกสารมาตรฐานไทย: ใบเสร็จ / ใบกำกับภาษีเต็มรูป / อย่างย่อ
- เลขที่เอกสารรันต่อเนื่องผ่าน doc_counters แบบ atomic กันซ้ำ (UPDATE ... RETURNING ในทรานแซกชัน)
- VAT 7% แยกบรรทัด + ยอดก่อน/หลังภาษี + หัก ณ ที่จ่าย (ตัวเลือก)
- เทมเพลตพิมพ์ PDF A4 และใบเล็ก 80mm พร้อมโลโก้ร้าน
```

## เฟส K — POS หน้าร้าน
เป้าหมาย: ขายหน้าร้านได้เร็ว ตัดสต๊อก + ปิดยอดรายวัน

ขอบเขต
- หน้าจอขายเร็ว: เลือกสินค้า/เมนู → ตะกร้า → ปรับจำนวน/ส่วนลด
- รับเงิน: เงินสด (คำนวณเงินทอน) · PromptPay QR (สร้าง QR ตามยอด) · บัตร (ถ้ามีเครื่อง EDC)
- พิมพ์ใบเสร็จย่อ 80mm + (ออปชัน) เปิดลิ้นชัก
- ตัดสต๊อกอัตโนมัติเมื่อขาย (เชื่อมระบบสต๊อก/ผลิตเดิม)
- เปิด/ปิดกะ (`pos_sessions`) + สรุปยอดรายวัน + สินค้าขายดี
- หลายแคชเชียร์ผูก shop_id เดียว (แยกข้อมูลที่ชั้น API)

ข้อจำกัด/อนาคต: ขายออฟไลน์เต็มรูป ต้องทำ cache/sync เพิ่ม — ทำภายหลัง

Prompt (Claude Code):
```
อ่าน backend/db/schema-extend.sql (pos_sessions + bills.payment_method)
ทำหน้า POS ใน frontend/pos/:
- ตะกร้าขายเร็ว + ส่วนลด + รับเงินสด(ทอน)/PromptPay QR
- บันทึกเป็น bill (doc_kind='pos') + ตัดสต๊อกอัตโนมัติ
- เปิด/ปิดกะ + สรุปยอดรายวัน + พิมพ์ใบเสร็จย่อ 80mm
```

## เฟส L — ฉลาก / แพ็กเกจ / จัดส่งพัสดุ
เป้าหมาย: ออกแบบฉลากแล้ว "สร้างไฟล์พร้อมพิมพ์ (PNG/PDF) ขนาดตรงมาตรฐาน ใช้ได้กับเครื่องพิมพ์ทุกยี่ห้อ" (รวม Niimbot และเครื่องพิมพ์ความร้อนทั่วไป) — แนวทางนี้ยืนยันแล้ว

ขอบเขต
- ฉลากสินค้า: ชื่อสินค้า · ราคา · บาร์โค้ด/QR · วันผลิต-หมดอายุ · ส่วนผสม/โภชนาการเบื้องต้น · โลโก้ร้าน
- ฉลากที่อยู่จัดส่ง: ผู้รับ/ผู้ส่ง · เบอร์โทร · ที่อยู่ · ยอด COD · เลขพัสดุ · โลโก้ขนส่ง
- ขนาดมาตรฐาน: 40×30 · 50×30 · 100×150 มม. (เก็บใน `label_templates.size`)
- **สร้างไฟล์พร้อมพิมพ์ PNG/PDF ที่ DPI ถูกต้อง** (เช่น 203 dpi สำหรับเครื่องพิมพ์ความร้อน) เพื่อให้ขนาดตรงเป๊ะและพิมพ์ได้กับทุกยี่ห้อ — รวม Niimbot (ผู้ใช้สั่งพิมพ์ไฟล์รูปผ่านแอป/ไดรเวอร์ของเครื่องเอง ไม่ต้องผูก SDK เฉพาะยี่ห้อ)
- เก็บเทมเพลตฉลากใน `label_templates` (layout_json) เพื่อแก้ไข/ใช้ซ้ำ
- บันทึกการจัดส่งใน `shipments` (เลือก courier · เลขพัสดุ · COD · สถานะ) เชื่อมกับ `bills`

ข้อจำกัด/อนาคต: เชื่อม API ขนส่ง (ไปรษณีย์ไทย/Flash/Kerry/J&T) เพื่อออกเลขพัสดุอัตโนมัติ + พิมพ์ใบปะหน้าจากระบบขนส่งโดยตรง — แยกเป็นเฟสถัดไป

Prompt (Claude Code):
```
อ่าน backend/db/schema-extend.sql (shipments + label_templates)
ทำโมดูลฉลากใน frontend/labels/:
- ออกแบบฉลากสินค้า + ฉลากที่อยู่จัดส่ง (เลือกขนาด 40x30 / 50x30 / 100x150 mm)
- เรนเดอร์เป็นไฟล์ PNG/PDF ขนาดตรง DPI พร้อมพิมพ์ (ใช้ได้กับ Niimbot + เครื่องความร้อนทั่วไป)
- บันทึกข้อมูลจัดส่งลง shipments + เก็บเทมเพลตใน label_templates
```
